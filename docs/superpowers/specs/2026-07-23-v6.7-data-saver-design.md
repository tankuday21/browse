# v6.7 — Data Saver (block images)

**Goal:** Let the user stop web pages from loading images to save mobile data — globally, with a
per-site override (block everywhere but allow a photo site, or vice-versa).

**Why:** On metered/slow connections images dominate page weight. A "Data saver" toggle is a
standard mobile-browser feature. Andromeda already has the per-site display-memory infrastructure
(H6: `site_settings` tri-state for forceDark/desktop/textZoom), so block-images slots straight in
as another tri-state — global default + per-site override — reusing `SiteSettingsResolver`.

## Design

`WebSettings.setBlockNetworkImage(true)` blocks network-loaded images (data: URIs still render).
That's exactly "data saver": no image bytes over the network.

### `data/SiteSettingsEntity.kt`
Add a fourth tri-state column: `blockImages: Int = -1` (-1 unset → fall through to global; 0 = show
images here; 1 = block here).

### Schema migration v20 → v21
- `@Database(version = 21)`, register `MIGRATION_20_21`.
- `ALTER TABLE site_settings ADD COLUMN blockImages INTEGER NOT NULL DEFAULT -1`.
- Export `21.json`; add a `MigrationTestHelper` androidTest 20→21 asserting an existing row's
  `blockImages` defaults to -1 and pre-existing columns survive.

### `browser/SiteSettingsResolver.kt`
`Effective` gains `blockImages: Boolean`. `resolve(...)` gains `globalBlockImages: Boolean` and sets
`blockImages = resolveTriState(override?.blockImages, globalBlockImages)`. (Same tri-state helper as
forceDark/desktop.)

### `data/SettingsRepository.kt`
Global toggle: `dataSaver: Flow<Boolean>` + `setDataSaver(enabled)`, key
`booleanPreferencesKey("data_saver")`, default **false** (images load out of the box). Fake mirror.

### `BrowserViewModel.kt`
- `dataSaver: StateFlow<Boolean>` (Eagerly, false) + `onDataSaverToggled(enabled)`.
- `onSetSiteOverride(...)` gains `blockImages: Int? = null`, merged like the other fields; the
  "all fields -1 ⇒ delete the row" check includes `blockImages`.

### `ui/WebViewHolder.kt`
- `@Volatile globalBlockImages: Boolean = false`.
- `applySiteSettings` (page start) passes `globalBlockImages` to `resolve` and applies
  `webView.settings.blockNetworkImage = effective.blockImages`.
- `obtain()` sets `settings.blockNetworkImage` from the resolved value for the fresh WebView.
- `applyGlobalBlockImages(enabled)`: sets the field and re-applies to every open tab (re-resolving
  each against its site override), mirroring `applyGlobalTextScale`. (Already-rendered images
  don't retroactively appear on unblock until the next load/reload — documented, minor.)
- Live per-site apply for the sheet: `applyBlockImages(tabId, enabled)` sets that tab's
  `blockNetworkImage` immediately (mirrors `applyForceDark(tabId, …)`).

### `MainActivity.kt`
`LaunchedEffect(dataSaver) { holder.applyGlobalBlockImages(dataSaver) }`, mirroring the `textScale`
wiring.

### UI
- **Settings screen:** a "Data saver" switch ("Block images to save data") in the browsing/privacy
  group, wired to `dataSaver`/`onDataSaverToggled`.
- **Site-settings sheet** (BrowserMenuSheet, where forceDark/desktop/zoom live): a "Block images"
  control with the same tri-state affordance the siblings use (default / on / off), calling
  `onSetSiteOverride(blockImages = …)` and the live `applyBlockImages(tabId, …)`.

### Incognito
`onSetSiteOverride` already returns for incognito (no persisted row); the sheet's live apply still
affects the visible tab. Unchanged.

## Testing
- `SiteSettingsResolverTest`: `blockImages` tri-state — override 1 wins over global false; override
  0 wins over global true; -1 falls through both ways.
- `BrowserViewModelOrbitTest`/settings: `onSetSiteOverride(blockImages=1)` persists; setting it back
  to -1 with the others already -1 deletes the row; `onDataSaverToggled` flips the flag.
- Migration androidTest 20→21.
- `FakeSettingsRepository` mirror.

## Non-goals
Blocking other resource types (video/fonts/CSS); a data-usage meter; automatic data-saver on
metered networks; lazy-load heuristics.
