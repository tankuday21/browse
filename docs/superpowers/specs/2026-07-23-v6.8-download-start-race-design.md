# v6.8 — DownloadService start-path race fix

**Goal:** Stop a download that the user cancels while it's queued/starting from leaving an orphan
file behind and/or resurrecting itself into a RUNNING state. This is the documented backlog item
("DownloadService start-path row re-check") first noted in v5.5.

## The bug

`DownloadService.handleStart(id)` (an ACTION_START/RESUME handler) launches a coroutine that:
1. `dao.getById(id)` → entry
2. creates the destination file (`createNewFile()`, to claim a collision-free name)
3. `dao.setState(id, "RUNNING")`
4. `engine.start(...)`

The ACTION_CANCEL handler runs concurrently: `engine.cancel(id)` + `dao.setState(id,"CANCELLED")`.
`engine.cancel(id)` is a **no-op when the engine isn't tracking `id` yet** (`generations[id] ?: return`),
and the engine only deletes the dest file when *it* is the one cancelled mid-download
(`DownloadEngine.runDownload` catch → `destFile.delete()`).

So if a CANCEL lands before/around `engine.start`:
- The empty file `handleStart` created is **orphaned** (nothing ever deletes it).
- `handleStart`'s `setState(RUNNING)` can **overwrite** the CANCEL's `setState(CANCELLED)`, so the
  download the user cancelled becomes an "undead" RUNNING transfer.

## The fix

Two changes in `DownloadService.handleStart`, plus one atomic DAO query:

1. **Early re-check (fast path).** After `getById`, before creating any file, abort if the row is
   already terminal-cancelled: `if (entry.state == "CANCELLED") { cleanup+return }`. This closes
   the common case — a cancel of a still-queued download — with no file ever created.

2. **Atomic claim (closes the narrow window).** Replace the unconditional `dao.setState(id,"RUNNING")`
   with an atomic conditional transition:
   ```
   @Query("UPDATE downloads SET state = 'RUNNING' WHERE id = :id AND state != 'CANCELLED' AND state != 'DONE'")
   suspend fun markRunningIfLive(id: Long): Int   // rows affected: 1 = claimed, 0 = cancelled/done/gone
   ```
   SQLite serializes writes, so this transition can't interleave with the CANCEL's
   `setState(CANCELLED)` — exactly one wins. If `markRunningIfLive` returns 0, the download was
   cancelled (or completed, or the row vanished) after our file setup, so **abort and delete the
   dest file if we created it this run** (matching the engine's own cancel-deletes-file semantics),
   then decrement `activeCount` and `stopIfIdle()`. Only call `engine.start` when the claim succeeds.

   Excluding `DONE` also prevents a stray START from resurrecting a completed download.

### Residual (documented)

A CANCEL that lands in the sliver between `markRunningIfLive` succeeding and `engine.start`
registering the generation is still possible; there the engine's own cancel path takes over on the
next check and deletes the file. This fix closes the dominant, user-visible window (cancel a
queued/scheduled download) deterministically at the DB layer; fully serializing START vs CANCEL
per-id (a mutex) is deferred as unnecessary for the observed bug.

## Files
- `data/DownloadDao.kt`: add `markRunningIfLive`.
- `download/DownloadService.kt`: early re-check + atomic claim + orphan-file cleanup in
  `handleStart`.
- `FakeDownloadDao`: mirror `markRunningIfLive`.

## Testing
- `FakeDownloadDao` JVM test (documents the claim contract): live PENDING → claim returns 1 and
  state becomes RUNNING; CANCELLED → returns 0, state stays CANCELLED; DONE → returns 0, stays DONE;
  missing id → 0.
- androidTest `downloadDao_markRunningIfLive` on real Room: same four cases (the authoritative
  check of the SQL `WHERE` clause; runs on-device).

Note: `handleStart` itself lives in an Android `Service` and isn't JVM-unit-testable; the atomic
claim is the mechanism under test, and its DB behavior is covered by the DAO tests. Full race
timing needs on-device verification (cancel a large queued download, confirm no leftover file in
the app's external Downloads dir and the row stays CANCELLED).

## Non-goals
Per-id start/cancel mutex; reworking the engine's file lifecycle; changing the retry/resume paths.
