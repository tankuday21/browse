# v6.6 — "Never save password for this site"

**Goal:** Let the user tell Andromeda to stop offering to save a login for a given site. This is
the explicitly-deferred companion to the save prompt — `SavePasswordBar`'s own KDoc says *"a
persistent 'never for this site' list is a later phase."*

**Why:** On sites where the user deliberately doesn't want a password stored (shared kiosks,
throwaway accounts, a work SSO they must always type), today's only option is to dismiss the save
bar every single time. Chrome/Firefox offer a "Never" choice that remembers the decision.

## Design

A per-app set of hosts (not per-Orbit: "never offer to save here" is a UX preference about a
site, and the save prompt is already Orbit-agnostic in how it's triggered). Stored as a
DataStore string-set, mirroring `adAllowedSites`.

### `data/SettingsRepository.kt`
```
val neverSaveSites: Flow<Set<String>>          // default emptySet
suspend fun toggleNeverSaveSite(host: String)  // add if absent, remove if present
suspend fun removeNeverSaveSite(host: String)   // management-screen delete
```
Key: `stringSetPreferencesKey("never_save_sites")`. Hosts stored already-normalized (lowercase
via `UrlHosts`). Fake mirrors the same in `FakeSettingsRepository`.

### `BrowserViewModel.kt`
- Expose `neverSaveSites: StateFlow<Set<String>>` (stateIn, initial emptySet).
- **Gate** in `onLoginSubmitted`: after computing `normHost`, if `normHost in
  neverSaveSites.value` → return without setting `_saveCredentialPrompt`. (Reading the StateFlow
  value is synchronous and safe here; the flow is warmed at VM init.)
- `onNeverSaveForSite()`: reads the pending `SaveCredentialPrompt`, persists its host via
  `toggleNeverSaveSite` (guarded so it only ever ADDS from this path — see note), and clears the
  prompt. Because the prompt is already showing, the host is by definition not yet in the set, so
  a toggle adds it; to be robust against a double-tap, use an explicit add rather than toggle from
  this entry point → add a repo `addNeverSaveSite` OR check membership before toggling. **Decision:
  the repo exposes `toggleNeverSaveSite` for the management screen; `onNeverSaveForSite` checks
  `host !in neverSaveSites.value` before calling it**, so it can only add.

### `ui/components/CredentialPrompts.kt` — `SavePasswordBar`
Add a third action, "Never", between Save and the dismiss X:
`Save` | `Never` | ✕. `onNever` wires to `onNeverSaveForSite`. Keep the row from overflowing on
narrow screens — the host text already has `weight(1f)` + ellipsis, and three compact TextButtons
fit (verify visually; if tight, the two text buttons can drop to icon-only later — not now).

### `ui/PasswordsScreen.kt` — management
Add a "Never saved on" section (only when the set is non-empty) listing each host with a delete
(trash) affordance calling `removeNeverSaveSite`. This is the only way to undo a "Never", so it
must exist. Place it below the saved-credentials list, using the existing `OrbitListRow` style.

## Interaction with v6.5

Independent. v6.5 broadened *fill* matching; v6.6 suppresses the *save* prompt. A never-saved
host simply never produces a `SaveCredentialPrompt`; nothing about fill changes.

## Testing (`BrowserViewModelOrbitTest` + settings)
- A login submitted on a never-saved host produces NO save prompt.
- A login on a normal host still prompts (positive control).
- `onNeverSaveForSite` on a pending prompt: host is added to the set AND the prompt is cleared.
- Calling it again (host already in set) does not duplicate / does not crash (add-only guard).
- `removeNeverSaveSite` drops the host, re-enabling the prompt on the next submit.
- Incognito still never prompts (unchanged), regardless of the never-save set.

## Non-goals
Per-Orbit never-save scoping; wildcard/subdomain never-save (exact host only, matching how the
save prompt keys today); auto-import of a "never" list.
