# v6.5 — Cross-Subdomain Password Fill

**Goal:** Offer a saved login for `example.com` when the user is on `login.example.com` (or
any host under the same registrable domain), instead of only on the exact host it was saved
under.

**Why:** Sites commonly present their login form on a subdomain (`accounts.google.com`,
`login.microsoftonline.com`, `signin.aws.amazon.com`) while the user saved the credential on the
bare domain — or vice-versa. Today's exact-host match (`getForOrbitAndHost`) silently offers
nothing, which reads as "the browser lost my password." This is the documented Passwords Phase-2
backlog item ("cross-subdomain fill").

## Security model (this is the whole design)

Filling the wrong site's password is the failure that matters, so matching is deliberately
conservative and every existing gate is preserved:

1. **Registrable-domain EQUALITY, not suffix containment.** Two hosts match iff they reduce to
   the *same* registrable domain (eTLD+1). We never test "is A a suffix of B", which is the
   classic hole (`example.com` must not match `example.com.evil.net`). By comparing the computed
   registrable domains for equality:
   - `example.com` vs `login.example.com` → both `example.com` → **match**
   - `example.com` vs `example.com.evil.net` → `example.com` vs `evil.net` → **no match**
   - `example.com` vs `notexample.com` → **no match**
2. **Public-suffix awareness.** A curated public-suffix set (common multi-label ccTLDs like
   `co.uk`, `com.au`, `co.in`, plus site-hosting suffixes like `github.io`, `web.app`,
   `pages.dev`, `herokuapp.com`) means `example.co.uk` and `other.co.uk` reduce to *different*
   registrable domains, and `a.github.io`/`b.github.io` do NOT match. The set is a curated
   subset of the Mozilla PSL — see Residual limitation.
3. **HTTPS-only, unchanged.** Fill is still offered only on `https://` pages (no injecting a
   credential into a MITM-able cleartext page).
4. **User-tap-initiated, unchanged.** The fill bar only *offers* usernames; the password is
   injected on the user's explicit tap, never automatically. So even a hypothetical over-match
   (a public suffix not in our curated set) can only surface an extra suggestion the user is free
   to ignore — it can never silently leak a password.
5. **Per-Orbit isolation, unchanged.** Matching runs within a single Orbit's vault; the offer is
   built for the Orbit captured at page-finish time.
6. **Incognito, unchanged.** Capture/fill bridges are withheld from incognito.

## Residual limitation (documented, accepted)

The public-suffix set is curated, not the full Mozilla list (~9,000 entries, ~230 KB). **Any
per-tenant public suffix we don't carry causes two different owners on that suffix to reduce to the
same registrable domain and over-match** — this is not limited to rare/obscure hosts. The curated
set therefore explicitly includes the common multi-tenant SaaS suffixes (`myshopify.com`,
`zendesk.com`, `atlassian.net`, `sharepoint.com`, `notion.site`, `webflow.io`, `firebaseio.com`,
`r2.dev`, …) so mainstream cases stay isolated. A suffix still absent from the set over-matches,
but two independent mitigations hold in code and BOTH must fail for a password to leak:
(1) fill is HTTPS-only and user-tap-initiated — never auto-injected (property 3–4); and (2) when a
candidate's stored host differs from the page host, the fill button renders a muted `@host` hint,
so a foreign-tenant suggestion visibly names the wrong host. The consequence of an over-match is
therefore at worst an extra, clearly-labelled, ignorable suggestion. Bundling the full Mozilla PSL
(which would close the class entirely) is deferred.

## Components

### `browser/CredentialHostMatch.kt` (pure, JVM-tested)

```
object CredentialHostMatch {
    // Curated public suffixes (multi-label). Single-label TLDs are implicit.
    private val PUBLIC_SUFFIXES: Set<String> = setOf("co.uk", "com.au", "github.io", ...)

    /** Lowercased registrable domain (eTLD+1), or null for blank/IP/single-label input. */
    fun registrableDomain(host: String?): String?

    /** True iff both hosts reduce to the same non-null registrable domain. */
    fun sameSite(a: String?, b: String?): Boolean

    /**
     * Candidate credential hosts that fill on [pageHost], ranked: exact-host first, then
     * same-registrable-domain, dropping non-matches. Stable within a rank tier.
     */
    fun rankHosts(pageHost: String, credentialHosts: List<String>): List<String>
}
```

Rules for `registrableDomain`:
- Lowercase, strip a trailing dot.
- Reject blank, and reject bare IPv4 / bracketed IPv6 literals (no registrable domain concept).
- Split into labels. Find the longest trailing label-sequence present in `PUBLIC_SUFFIXES`
  (multi-label) — if found and there is at least one label before it, registrable = that suffix
  plus one preceding label. Otherwise treat the last label as the TLD: registrable = last two
  labels. If the host has only one label (or fewer labels than the matched suffix + 1), return
  null (can't form a registrable domain — don't match anything).

### `data/CredentialDao.kt`

Add a suspend one-shot to read a whole Orbit's rows (plaintext host is enough to filter before
decrypting — we only decrypt the matches):

```
@Query("SELECT * FROM credentials WHERE orbitId = :orbitId ORDER BY updatedAt DESC")
suspend fun getAllForOrbit(orbitId: Long): List<CredentialEntity>
```

### `data/CredentialRepository.kt`

```
/**
 * Decrypted logins whose stored host fills on [pageHost] (same registrable domain), ranked
 * exact-host-first. Only matching rows are decrypted. Rows that fail to decrypt are dropped.
 */
suspend fun credentialsForSite(orbitId: Long, pageHost: String): List<DecryptedCredential>
```

Implementation: `getAllForOrbit` → filter `CredentialHostMatch.sameSite(pageHost, row.host)` →
order by `rankHosts(pageHost, matchedHosts)` (exact first) then existing recency → decrypt.

### `BrowserViewModel.kt`

- `FillPromptState.usernames: List<String>` → `candidates: List<FillCandidate>` where
  `FillCandidate(host: String, username: String)`. The stored host must ride along so the fill
  re-lookup targets the credential's OWN host, not the page host.
- Page-finish fill offer: replace `credentialsForHost(orbitId, host)` with
  `credentialsForSite(orbitId, host)`; dedup by username keeping the highest-ranked host (so a
  username saved on both bare and www shows once, preferring the exact-host row).
- `onFillCredential(tabId, orbitId, host, username)` unchanged in shape — `host` is now the
  candidate's stored host, re-looked-up via the existing exact `credentialsForHost`.

### `ui/components/CredentialPrompts.kt`

`FillPasswordBar` takes `candidates: List<FillCandidate>` and calls `onFill(candidate)`. Button
label stays the username; when a candidate's host differs from the page host, append a muted
`@host` hint so the user can tell a subdomain credential apart. Still capped at 3 buttons.

## Testing

`CredentialHostMatchTest` (pure):
- exact host, bare↔subdomain both directions, deep subdomain
- `example.com` vs `example.com.evil.net` (no), `notexample.com` (no), `evilexample.com` (no)
- multi-label suffix: `a.co.uk` vs `b.co.uk` (no), `x.a.co.uk` vs `a.co.uk` (yes)
- site-hosting suffix: `a.github.io` vs `b.github.io` (no)
- IPv4 / IPv6 literal / single-label → null / no match
- trailing-dot and mixed-case normalization
- `rankHosts` puts exact first, drops non-matches, stable within tier

`BrowserViewModelTest` additions:
- credential saved on `example.com` → fill offered on `https://login.example.com`
- credential saved on `login.example.com` → offered on `https://example.com`
- no offer on `https://example.com.evil.net`
- no offer on cleartext `http://login.example.com` (HTTPS gate preserved)
- dedup: same username on `example.com` + `www.example.com` → one candidate, host = `example.com`
- fill uses the candidate's stored host (regression: injecting the right password)

## Non-goals

Full Mozilla PSL bundling; "never for this site" list; fuzzy/typo host matching; filling across
different registrable domains (federated-login linking).
